## 2026-05-02   现在支持以下 Google Pixel 录音相关的开关：
1. ✅ 开启通话录音 - 主要开关
2. ✅ 自动通话录音 - 自动录音开关
3. ✅ 自动录制与非通讯录号码的通话 - 更细粒度的开关

### 1. 删除手动拨号和通讯录功能
- 删除文件：`ContactDao.kt`、`ContactRepository.kt`、`ContactViewModel.kt`、`AddEditContactScreen.kt`、`CallSpeakerHelper.kt`、`CallStateMonitorService.kt`
- 清理范围：`ContactEntity`、通讯录路由、手动/自动模式切换、`ACTION_DIAL_NEXT`/`ACTION_SET_AUTO_MODE`、悬浮窗手动拨号按钮
- 数据库版本升级：v6 → v7（删除 contacts 表）

### 2. 修复 foregroundServiceType 权限问题
- 问题：`phoneCall` 类型需要 `MANAGE_OWN_CALLS` 权限或默认拨号应用角色
- 修复：`foregroundServiceType` 从 `phoneCall` 改为 `specialUse`，删除 `FOREGROUND_SERVICE_PHONE_CALL` 权限

### 3. 实现拨号后自动扩音
- `CallHelper.enableSpeakerphone()`：先设置 `MODE_IN_CALL`，再调用 `setCommunicationDevice()`（Android 12+）或 `setSpeakerphoneOn`，最后拉满音量
- `AutoDialService`：在 `CALL_STATE_OFFHOOK` 时延迟 800ms 后开启免提，最多重试 3 次

### 4. 任务客户分页加载（支持无限客户数）
- 后端：新增 `GET /tasks/:id/customers` 分页接口
- Android：`TaskRepository.getAllPendingTaskCustomers()` 自动分页加载所有待拨打客户
- 默认每页 100 条，循环加载直到获取全部

### 5. 修复自动拨号语音信箱误判和前台检查问题
- **问题**：语音信箱播报时通话状态短暂变为 IDLE，导致误判通话结束；前台检查时机错误导致后台时仍拨打下一个
- **修复**：
  - 新增 `MIN_CONFIRM_IDLE_TIME = 3000ms`，必须持续 IDLE 状态 3 秒才确认通话结束
  - 状态从 IDLE 变回 OFFHOOK 时重置确认计时（处理语音信箱播报期间状态变化）
  - 修正前台检查时机：`通话结束 → delay(间隔) → waitUntilCanDial() → 拨打下一个`
- **效果**：App 必须处于前台才能拨打下一个号码

### 6. 修复前台监控逻辑
- 问题：原逻辑 `!isInForeground || !isCallIdle` 在通话中时错误挂起
- 修复：改为 `!isInForeground && isCallIdle`，只在"不在前台且通话空闲"时挂起

### 7. Root 检测器降级处理
- 非 Root 设备跳过精确通话状态检测，使用 TelephonyManager 基础状态

### 8. 修复大量客户时 Intent 数据超限问题
- **问题**：Intent 数据大小限制约 1MB，1000+ 客户序列化后超限导致启动失败
- **修复**：
  - 新增 `EXTRA_LOAD_FROM_TASK` 标志，支持任务加载模式
  - 任务加载模式：只传 taskId，在 Service 中分页加载客户
  - 直接传递模式：客户数 ≤ 100 时仍通过 Intent 传递（兼容旧逻辑）
- **涉及文件**：`AutoDialService.kt`、`AutoDialViewModel.kt`、`SequentialAutoDialViewModel.kt`

### 9. 代码优化 - 提取常量和免提方法
- 将通话状态检测常量提取到 `companion object`：
  - `MIN_CALL_DURATION`、`MAX_CALL_DURATION`、`MIN_CONFIRM_IDLE_TIME`
  - `SPEAKER_ENABLE_DELAY`、`SPEAKER_RETRY_COUNT`、`SPEAKER_RETRY_DELAY`
- 提取免提开启逻辑为独立方法 `enableSpeakerphoneWithRetry()`
- 移除未使用的 `elapsedTime` 变量

### 10. 优化通话状态识别逻辑
- **问题**：之前只要进入 OFFHOOK 且通话 >= 3秒就判断为"已接听"，导致语音信箱被误判
- **修复**：增加 `VOICE_MAIL_THRESHOLD = 15000ms`（15秒）阈值
- **判断逻辑**：
  | 状态 | 判断条件 |
  |---|---|
  | 用户本人接听 | OFFHOOK + 通话时长 >= 15秒 |
  | 语音信箱 | OFFHOOK + 通话时长 3-15秒 |
  | 响铃未接 | 从未进入 OFFHOOK |
- **历史记录优化**：进入 OFFHOOK 时记录"已进入通话"，通话结束后根据时长判断最终结果

### 11. 恢复完整扬声器启用策略（6层防护 + 17次重试）
- **问题**：之前简化后的实现缺少反射调用，导致非 Root 设备扬声器启用不稳定
- **修复**：恢复完整的多阶段启用策略
- **6层防护策略**：
  | Phase | 方法 | 说明 |
  |---|---|---|
  | Phase 1 | `MODE_IN_CALL` | 设置音频模式 |
  | Phase 2 | `isSpeakerphoneOn = true` | AudioManager API |
  | Phase 3 | 音量拉满 | 通话音量最大 |
  | Phase 4 | `AudioSystem.setForceUse` 反射 | **关键！非Root也能用** |
  | Phase 5 | `setCommunicationDevice` | Android 12+ 新API |
  | Phase 6 | Root 命令 | 可选，Root设备增强 |
- **17次重试机制**：间隔 50ms、100ms、150ms... 直到 850ms
- **非 Root 兼容**：Phase 1-5 都不需要 Root 权限

### 12. 恢复 InCallService 实现稳定自动扩音
- **核心原理**：`InCallService.setAudioRoute()` 是电信框架级API，优先级最高
- **为什么之前不稳定**：
  1. 只在 `onCallAdded` 设置一次，系统后续重置无法修复
  2. 没有持续监控机制
  3. 没有处理 `onCallAudioStateChanged` 回调
- **新实现的关键设计**：
  | 时机 | 操作 |
  |---|---|
  | `onCallAdded` | 立即设置扬声器（最早时机）|
  | `onCallAudioStateChanged` | 检测路由被重置，立即修复 |
  | `Call.STATE_DIALING` | 拨号中，再次设置 |
  | `Call.STATE_ACTIVE` | 通话接通，启动持续监控 |
  | 持续监控 | 每 500ms 检查扬声器状态，被关闭则重新设置 |
- **必要条件**：应用必须设置为默认拨号应用
- **新增文件**：
  - `AutoSpeakerInCallService.kt` - InCallService 实现
  - `DefaultDialerHelper.kt` - 默认拨号应用请求工具
  - `DialerActivity.kt` - 拨号入口（成为默认拨号应用的必要条件）
- **设置页面**：添加"默认拨号应用"设置项
- **小米手机兼容**：添加 `DialerActivity` 处理 `ACTION_DIAL`，确保能出现在默认拨号应用列表中

- 验证结果：已执行 `./gradlew :app:compileDebugKotlin`，编译通过。

## 2026-04-15 APK 下载地址 404 修复
- 本次完成：修复 `/admin/version` 返回的 APK 地址可用性问题，避免 `/uploads/apk/app-release-xxx.apk` 出现 404。
- 根因：版本创建、APK 上传和静态文件访问都只依赖单一 `uploads/apk` 目录；一旦线上运行目录与上传落盘目录不一致，版本接口能返回 URL，但静态资源找不到文件。
- 修复方式：服务端为 APK 增加多目录候选解析，版本创建时按候选目录查找文件，上传时统一写入可写目录，同时新增 `/uploads/apk/:fileName` 下载兜底路由，按多目录顺序查找并返回 APK。
- 验证结果：已执行 `pnpm -C /Users/bytedance/del/call-center-system/server build`，构建通过。
